# Research: Deterministic Daily Log Script Replacement

## Objective

Document every behavior of the daily-log skill (`SKILL.md`) in sufficient detail for Ryan to build a deterministic Python script (`bin/update-daily-log`) that replaces the LLM orchestration. This is the most complex skill in the project because it handles both creation and update paths, constructs a large batchUpdate with multiple formatting operations, and delegates to two sub-skills.

---

## Phase 1: The "Update" Path (Entry Already Exists)

When a HEADING_1 matching today's date is found in the document, the daily-log skill delegates entirely to two sub-skills. It does not build any batchUpdate itself.

### Flow

1. Run the task-list sub-skill (already replaced by `bin/update-task-list --auto-confirm`)
2. Run the open-tickets sub-skill (already replaced by `bin/update-open-tickets --auto-confirm`)
3. Report what each sub-skill updated

### API Calls

None made directly by the daily-log skill in this path. The sub-skill scripts make their own API calls:

- `update-task-list`: `gws-safe calendar events list`, `gws-safe tasks tasks list` (x2), `gws-safe docs documents get`, `gws-safe docs documents batchUpdate`, final `gws-safe docs documents get` for cache
- `update-open-tickets`: `freshdesk-safe tickets search`, `freshdesk-safe contacts view` (per unknown requester), `gws-safe docs documents get`, `gws-safe docs documents batchUpdate`, final `gws-safe docs documents get` for cache

### Implementation for Ryan

The update path in the new script is straightforward: call the two existing scripts in sequence.

```python
subprocess.run(["bin/update-task-list", "--auto-confirm"], check=True)
subprocess.run(["bin/update-open-tickets", "--auto-confirm"], check=True)
```

The sub-skills already handle their own caching, dry-run confirmation, and cache updates.

## Phase 2: The "Create New Entry" Path (No Entry Exists)

This is where all the complexity lives. When no HEADING_1 matching today's date is found, the skill creates the entire daily log entry from scratch.

### Step-by-Step Breakdown

#### 2a: Gather Data

Five data sources, fetched in parallel where possible:

**1. Calendar events**

```
gws-safe calendar events list --params '{"calendarId":"primary","timeMin":"<YYYY-MM-DD>T00:00:00<TZ_OFFSET>","timeMax":"<YYYY-MM-DD>T23:59:59<TZ_OFFSET>","singleEvents":true,"orderBy":"startTime"}'
```

- `TZ_OFFSET` from `date +%:z` (e.g., `-04:00`)
- Filter out events where `eventType == "workingLocation"`
- Filter out events whose `colorId` is in `config.excludeEventColorIds` (currently `["8"]`)
- Extract from each event: `summary`, `start.dateTime` or `start.date`, `description`, `htmlLink`
- For each event, extract Support Doc URL from `description` using regex: `Support Doc:\s*(https://docs\.google\.com/document/d/[^\s]+)`

**2. In-progress tasks**

```
gws-safe tasks tasks list --params '{"tasklist":"TG16TWFNTkNRblVkdHdhbQ","showAssigned":true}'
```

- Extract: `title`, `webViewLink`
- Preserve original list order (this is the sort order for tasks in the entry)

**3. Waiting tasks**

```
gws-safe tasks tasks list --params '{"tasklist":"ZGRjdFdTV19KWGkxVTdMbg","showAssigned":true}'
```

- Filter to only tasks where `due` matches today: `<YYYY-MM-DD>T00:00:00.000Z`
- Tasks with future or past due dates are excluded

**4. Open tickets**

```
freshdesk-safe tickets search --query "status:2"
```

- Returns `{"total": N, "results": [...]}`
- Extract each ticket's `id`, `subject`, `requester_id`
- For each unique `requester_id`, resolve name:
  - First check CRM database: `SELECT name FROM contacts WHERE freshdesk_id = ?`
  - If not found, call `freshdesk-safe contacts view <REQUESTER_ID>` and use `name` field
  - If contact lookup fails entirely, use "Unknown"

**5. Random fact**

- Generated by the LLM from its own knowledge about today's date
- 1-2 sentences, quirky or funny historical fact
- **For the deterministic script:** This is the one piece the LLM currently generates. Options:
  - Pre-populate a set of facts per day-of-year
  - Call an external API
  - Use a simple placeholder like "No random fact today"
  - Call the LLM for just this one piece (defeats the purpose)
  - **Recommendation:** Use a simple static message or skip this section. If Timothy values the random facts, a separate data file of 366 facts could be pre-generated.

#### 2b: Build the Entry Text

The entire entry is composed as a single text block. The exact structure is:

```
<Today's Date>\n
Task List\n
\U0001f3af Priorities:\n
(event) <event title> <formatted time>\n
(event) <next event> <formatted time>\n
(task) <task title>\n
(task) <next task title>\n
\u23f3 Waiting / Blockers:\n
<waiting task title>\n
[or "None" if no waiting tasks due today]\n
\U0001f3b2 Random Fact:\n
<fact text>\n
Open Tickets:\n
[<ticket-id>] <customer> - <subject> - https://miarec.freshdesk.com/a/tickets/<ticket-id>\n
[or "None" if no open tickets]\n
Thoughts / Ideas:\n
\n
Notes\n
<Task/Event 1 title>\n
<link URL if available>\n
<Task/Event 2 title>\n
<link URL if available>\n
```

**Line-by-line details:**

| Line | Content | Style |
|------|---------|-------|
| 1 | Today's date (e.g., `Thursday Apr 2, 2026`) | HEADING_1 |
| 2 | `Task List` | HEADING_2 |
| 3 | `\U0001f3af Priorities:` | NORMAL_TEXT (section label) |
| 4+ | `(event) <title> <time>` or `(task) <title>` | NORMAL_TEXT + bullets (default preset) |
| N | `\u23f3 Waiting / Blockers:` | NORMAL_TEXT (section label) |
| N+1+ | Waiting task titles or "None" | NORMAL_TEXT + bullets (default preset) |
| M | `\U0001f3b2 Random Fact:` | NORMAL_TEXT (section label) |
| M+1 | The random fact text | NORMAL_TEXT |
| P | `Open Tickets:` | HEADING_2 |
| P+1+ | `[<id>] customer - subject - URL` or "None" | NORMAL_TEXT + checkbox bullets |
| Q | `Thoughts / Ideas:` | HEADING_2 |
| Q+1 | Empty line | NORMAL_TEXT |
| R | `Notes` | HEADING_2 |
| R+1+ | Event/task titles as HEADING_3, each followed by a link URL line | HEADING_3 + NORMAL_TEXT |

**Time formatting:**
- Timed events: `h:mm AM/PM` (e.g., `9:00 AM`) -- use `strftime("%-I:%M %p")`
- All-day events: `all day`

**Priority item ordering (within the Priorities sub-section):**
1. All-day events first, sorted alphabetically by title
2. Timed events next, sorted by start time ascending
3. Tasks last, in their original list order from the API response

**Notes section ordering:**
Same ordering as priorities: all-day events (alpha), timed events (by start time), tasks (list order).

**Notes link URL logic:**
- For events with a `Support Doc:` URL in `description`: use the Support Doc URL
- For events without a Support Doc: use the event's `htmlLink`
- For tasks: use the task's `webViewLink` (if available)
- Link URL lines appear on the line immediately following the HEADING_3 title

#### 2c: Insert via batchUpdate

This is where the real complexity lives. The entire entry is inserted at document index 1 (top of document) via a single batchUpdate call.

### The batchUpdate Request Structure

The batchUpdate is a single JSON object with a `requests` array. Requests are processed sequentially by the API. The order matters because text insertions shift all subsequent indices.

**Critical rule:** Insert all text first in a single `insertText` request, then apply all formatting. This avoids index-shifting complications between insert operations.

#### Request 1: insertText

```json
{
  "insertText": {
    "location": {"index": 1},
    "text": "<entire entry text with newlines>"
  }
}
```

The text is the complete entry text block from 2b, including all newlines. A trailing newline is added at the end.

#### Request 2: updateParagraphStyle - NORMAL_TEXT on all paragraphs

```json
{
  "updateParagraphStyle": {
    "range": {"startIndex": 1, "endIndex": <end of all inserted text>},
    "paragraphStyle": {"namedStyleType": "NORMAL_TEXT"},
    "fields": "namedStyleType"
  }
}
```

This is required because text inserted at index 1 inherits the paragraph style of whatever was at index 1 before (which is typically the previous day's HEADING_1). Resetting everything to NORMAL_TEXT first, then applying specific styles, is the safest approach.

#### Requests 3+: updateParagraphStyle for heading paragraphs

Apply heading styles to specific line ranges. Each request targets the startIndex and endIndex of a specific line (including its trailing newline).

**HEADING_1** - the date line:
```json
{
  "updateParagraphStyle": {
    "range": {"startIndex": <date_line_start>, "endIndex": <date_line_end>},
    "paragraphStyle": {"namedStyleType": "HEADING_1"},
    "fields": "namedStyleType"
  }
}
```

**HEADING_2** - section headings (`Task List`, `Open Tickets:`, `Thoughts / Ideas:`, `Notes`):
```json
{
  "updateParagraphStyle": {
    "range": {"startIndex": <heading_start>, "endIndex": <heading_end>},
    "paragraphStyle": {"namedStyleType": "HEADING_2"},
    "fields": "namedStyleType"
  }
}
```

**HEADING_3** - each Notes sub-section title (event/task names under Notes):
```json
{
  "updateParagraphStyle": {
    "range": {"startIndex": <sub_heading_start>, "endIndex": <sub_heading_end>},
    "paragraphStyle": {"namedStyleType": "HEADING_3"},
    "fields": "namedStyleType"
  }
}
```

#### Bullet requests: createParagraphBullets

**Priority items and waiting/blocker items** use the default bullet preset:
```json
{
  "createParagraphBullets": {
    "range": {"startIndex": <first_bullet_start>, "endIndex": <last_bullet_end>},
    "bulletPreset": "BULLET_DISC_CIRCLE_SQUARE"
  }
}
```

**Open ticket items** use the checkbox bullet preset:
```json
{
  "createParagraphBullets": {
    "range": {"startIndex": <first_ticket_start>, "endIndex": <last_ticket_end>},
    "bulletPreset": "BULLET_CHECKBOX"
  }
}
```

Values from `config/doc_styles.json`:
- `bulletPresets.default` = `"BULLET_DISC_CIRCLE_SQUARE"`
- `bulletPresets.checkbox` = `"BULLET_CHECKBOX"`

#### Font styling: updateTextStyle

**All HEADING_1, HEADING_2, HEADING_3 paragraphs** get heading font:
```json
{
  "updateTextStyle": {
    "range": {"startIndex": <heading_start>, "endIndex": <heading_end>},
    "textStyle": {
      "weightedFontFamily": {"fontFamily": "Lexend"}
    },
    "fields": "weightedFontFamily"
  }
}
```

**All NORMAL_TEXT paragraphs** (section labels, bullet items, random fact text, link lines) get body font:
```json
{
  "updateTextStyle": {
    "range": {"startIndex": <text_start>, "endIndex": <text_end>},
    "textStyle": {
      "weightedFontFamily": {"fontFamily": "Roboto"}
    },
    "fields": "weightedFontFamily"
  }
}
```

**Color handling:** Currently `colors.headingText` and `colors.bodyText` are both `null` in the style config. If they were non-null, the `updateTextStyle` requests would include `foregroundColor` in the `textStyle` object and add `"foregroundColor"` to the `fields` mask. The script should check for non-null and conditionally include them.

Values from `config/doc_styles.json`:
- `fonts.heading` = `"Lexend"`
- `fonts.body` = `"Roboto"`
- `colors.headingText` = `null` (no color override)
- `colors.bodyText` = `null` (no color override)

#### Link styling on Notes URL lines

After all the font styling, for each HEADING_3 that has a link URL, a NORMAL_TEXT line with the URL text was already inserted in the text block. Now apply link styling:

**Body font on the link line:**
```json
{
  "updateTextStyle": {
    "range": {"startIndex": <link_line_start>, "endIndex": <link_line_end>},
    "textStyle": {
      "weightedFontFamily": {"fontFamily": "Roboto"}
    },
    "fields": "weightedFontFamily"
  }
}
```

**Hyperlink on the URL text** (excluding trailing newline):
```json
{
  "updateTextStyle": {
    "range": {"startIndex": <link_text_start>, "endIndex": <link_text_end>},
    "textStyle": {
      "link": {"url": "<LINK_URL>"}
    },
    "fields": "link"
  }
}
```

### Index Calculation Algorithm

This is the most error-prone part. The algorithm:

1. All text is inserted at index 1 as a single string.
2. After insertion, the inserted text occupies indices `[1, 1 + utf16_len(text))`.
3. Each line's start index is computed by summing the UTF-16 lengths of all preceding lines (including their `\n` characters).
4. A newline `\n` counts as 1 UTF-16 code unit.

**UTF-16 length function** (from existing scripts):
```python
def utf16_len(text):
    """Google Docs API uses UTF-16 code unit indices."""
    return len(text.encode('utf-16-le')) // 2
```

**Index tracking pattern** (used in both existing scripts):
```python
cursor = 1  # insertion point
for line in lines:
    line_with_newline = line + "\n"
    line_start = cursor
    line_end = cursor + utf16_len(line_with_newline)
    # Use line_start and line_end for formatting requests
    cursor = line_end
```

**Important:** For plain ASCII text, `utf16_len(text) == len(text)`. But emojis and non-BMP characters take 2 UTF-16 code units. The emoji characters used in section labels are:
- `\U0001f3af` (Target / Bullseye) -- 2 UTF-16 code units
- `\u23f3` (Hourglass) -- 1 UTF-16 code unit
- `\U0001f3b2` (Die / Game Die) -- 2 UTF-16 code units

This means `utf16_len("🎯 Priorities:\n")` != `len("🎯 Priorities:\n")`. The `utf16_len` function handles this correctly.

### Complete Request Ordering

The full batchUpdate `requests` array, in order:

1. `insertText` at index 1 (single request, all text)
2. `updateParagraphStyle` NORMAL_TEXT on entire inserted range
3. `updateParagraphStyle` HEADING_1 on date line
4. `updateParagraphStyle` HEADING_2 on each of: Task List, Open Tickets:, Thoughts / Ideas:, Notes
5. `updateParagraphStyle` HEADING_3 on each Notes sub-section title
6. `createParagraphBullets` on priority items range (default preset)
7. `createParagraphBullets` on waiting/blocker items range (default preset)
8. `createParagraphBullets` on open ticket items range (checkbox preset)
9. `updateTextStyle` with heading font on all HEADING_1/2/3 ranges
10. `updateTextStyle` with body font on all NORMAL_TEXT ranges
11. `updateTextStyle` with body font on each link URL line
12. `updateTextStyle` with hyperlink on each link URL text (excluding trailing newline)

### Post-Insert Verification

After the batchUpdate succeeds, the skill re-reads the document and verifies:
- The date heading is styled as HEADING_1
- The Thoughts / Ideas and Notes lines are styled as HEADING_2
- Task/event sub-headings are styled as HEADING_3
- Priority and waiting items are bulleted

If formatting is incorrect, the error is reported (not auto-fixed).

### Post-Insert Cache Population

After successful verification, the cache is populated with ALL entries in the document (not just today's):

```
daily-log-cache populate --json '{"document_id":"<DOC_ID>","revision_id":"<REVISION_ID>","entries":[...]}'
```

The entries array includes every HEADING_1 entry in the document, each with its sections (HEADING_2) and subsections (HEADING_3). This is the same `parse_doc_entries()` function used in both existing scripts.

## Phase 3: Document Structure

### Entry Structure (Exact)

Each daily log entry has this fixed structure of sections:

```
[HEADING_1] Thursday Apr 2, 2026
[HEADING_2] Task List
[NORMAL_TEXT] 🎯 Priorities:
[NORMAL_TEXT, bulleted] (event) Event Title 9:00 AM
[NORMAL_TEXT, bulleted] (event) Another Event 2:30 PM
[NORMAL_TEXT, bulleted] (task) Task Title
[NORMAL_TEXT] ⏳ Waiting / Blockers:
[NORMAL_TEXT, bulleted] Waiting task title
[NORMAL_TEXT] 🎲 Random Fact:
[NORMAL_TEXT] Some interesting fact about today
[HEADING_2] Open Tickets:
[NORMAL_TEXT, checkbox-bulleted] [12345] Customer - Subject - https://miarec.freshdesk.com/a/tickets/12345
[NORMAL_TEXT, checkbox-bulleted] [12346] Customer2 - Subject2 - https://miarec.freshdesk.com/a/tickets/12346
[HEADING_2] Thoughts / Ideas:
[NORMAL_TEXT] (empty line for user to type in)
[HEADING_2] Notes
[HEADING_3] Event Title
[NORMAL_TEXT] https://docs.google.com/document/d/abc123 (link URL)
[HEADING_3] Task Title
[NORMAL_TEXT] https://mail.google.com/tasks/... (link URL)
```

### Section Types (Cache Schema)

The `sections` table uses these `section_type` values:

| section_type | Heading Text | Style |
|-------------|-------------|-------|
| `task_list` | `Task List` | HEADING_2 |
| `open_tickets` | `Open Tickets:` | HEADING_2 |
| `thoughts_ideas` | `Thoughts / Ideas:` | HEADING_2 |
| `notes` | `Notes` | HEADING_2 |

### Section Boundary Definitions

- Each section spans from its HEADING_2 `startIndex` to the next HEADING_2 `startIndex` (or the entry's end)
- Each entry spans from its HEADING_1 `startIndex` to the next HEADING_1 `startIndex` (or the document end)
- Subsections (HEADING_3) are only tracked under the `notes` section

### Date Format

- Heading text: `DayOfWeek Mon DD, YYYY` (e.g., `Thursday Apr 2, 2026`)
- Generated with: `date +'%A %b %-d, %Y'` or `datetime.now().strftime("%A %b %-d, %Y")`
- Parsed with: `datetime.strptime(text, "%A %b %d, %Y")` -- `%d` handles both padded and unpadded days

## Existing Patterns (Reusable Code from update-open-tickets and update-task-list)

### Shared Infrastructure

Both existing scripts share identical implementations of these functions. The new script should reuse them (either by importing from a shared module or copy-pasting).

**File I/O:**
- `load_json_file(path)` -- reads and parses a JSON config file
- `run_cmd(args, check=True)` -- runs a subprocess, returns (stdout, stderr, returncode), exits on error

**Database access:**
- `get_cache_db()` -- opens `db/daily_log_cache.db`, creates from schema if needed, returns connection
- Direct SQLite access to the cache (no need to shell out to `daily-log-cache` wrapper)

**Document parsing** (identical in both scripts):
- `parse_doc_entries(doc)` -- parses all HEADING_1 entries with sections and subsections
- `parse_date_from_heading(heading_text)` -- parses `"Friday Mar 27, 2026"` into `"2026-03-27"`
- `parse_entry_sections(body_content, entry_start, entry_end)` -- parses HEADING_2 sections
- `parse_subsections(body_content, section_start, section_end)` -- parses HEADING_3 subsections
- `SECTION_MAP` dict: `{"Task List": "task_list", "Open Tickets:": "open_tickets", ...}`

**Text extraction:**
- `extract_section_text(doc, section_start, section_end)` -- extracts plain text from a doc range

**Index calculation:**
- `utf16_len(text)` -- computes UTF-16 code unit length for Google Docs API indices

**batchUpdate execution:**
- `execute_batch_update(doc_id, batch_json, auto_confirm)` -- handles dry-run/nonce/auto-confirm flow

**Cache update:**
- `update_cache(doc_id)` -- re-reads doc, parses all entries, calls `daily-log-cache populate`

### batchUpdate Construction Pattern

Both scripts follow this pattern for building batchUpdate requests:

1. Build "insertion groups" -- each group is `(insert_index, text, formatting_fn)`
2. Sort groups by `insert_index` descending (highest first)
3. For each group:
   - Emit an `insertText` request
   - Call `formatting_fn(base_index, text)` to get formatting requests
   - Append formatting requests to the request list

The descending sort is critical: when making multiple insertions, higher-index insertions must go first so they don't shift lower-index insertion points.

For the create path, this pattern simplifies to a single insertion group at index 1 (since all text goes in one `insertText` call).

### Data Fetching with Caching (from update-task-list)

The `update-task-list` script already has cached fetch functions:
- `fetch_calendar_events(config, today_str, tz_offset)` -- uses `calendar_events_cache` table, 5-min TTL
- `fetch_tasks(config, today_str)` -- uses `tasks_cache` table, 5-min TTL, returns `(in_progress, waiting)`

The `update-open-tickets` script has:
- `fetch_open_tickets()` -- uses `open_tickets_snapshot` table, 5-min TTL, resolves requester names via CRM then FreshDesk API

**These can be reused directly by the create path.** The new script should call the same cached fetch functions so that if the create path runs and then immediately delegates to the update sub-skills, the sub-skills will hit the cache and avoid redundant API calls.

## Step 5: Archive Old Entries

The daily-log skill also handles archiving entries from before the current week. This is the second major complexity area. Key details:

### When Archiving Triggers
- After Step 3 (create) or Step 4 (update), the skill checks for entries older than the Monday of the current week
- "Monday of the current week" means: if today is Monday, it's today; otherwise, the most recent Monday

### Archive Document Structure
- Entries are archived to a separate Google Doc (`archiveDocumentId` in config)
- The archive doc uses a tab hierarchy: Year > Quarter > Week
- Tab titles: `2026`, `2026-Q1`, `Week 3 (Jan 19-23)`
- If Monday and Friday of a week span different months, the week tab format changes: `Week 10 (Sep 29 - Oct 3)`

### Tab Creation
- Missing tabs are created in dependency order: Year first, then Quarter (as child of Year), then Week (as child of Quarter)
- Each level is batched into a single `batchUpdate` with `addDocumentTab` requests
- Quarter index within year: Q1=0, Q2=1, Q3=2, Q4=3
- Week number is calculated relative to the quarter's first Monday

### Entry Insertion into Archive
- Entries are inserted at index 1 of the target week tab (newest first)
- The insertion must handle rich elements: `textRun`, `inlineObjectElement` (images), `richLink` (file chips), `person` (person chips)
- `richLink` elements are 1 character in source but expand to full title text in archive (no `insertRichLink` API exists)
- `person` elements are 1 character in both source and destination, inserted via `insertPerson`
- `inlineObjectElement` is 1 character, inserted via `insertInlineImage`
- Image and person insertions are sorted by index descending and placed after all other requests

### Deletion from Main Doc
- Only after successful archive insertion
- Uses `deleteContentRange` requests, processed bottom-to-top (highest indices first)
- Cache is invalidated and repopulated after deletion

### Recommendation for the New Script
The archive functionality is complex and involves multiple batchUpdate calls across two documents, tab management, and rich element handling. **Recommendation:** Defer archiving to a later phase. The create and update paths are where the daily usage cost is. Archiving runs at most once per week and is less time-sensitive.

## Risks and Caveats

1. **Random fact generation.** The LLM currently generates this from its knowledge. A deterministic script cannot do this without an external source. This is a minor feature -- the script can use a placeholder or skip it. Confidence: high that this is acceptable.

2. **UTF-16 index arithmetic.** The emoji characters in section labels (`🎯`, `⏳`, `🎲`) occupy 2 UTF-16 code units each (except `⏳` which is 1). The `utf16_len()` function handles this correctly. Every string length calculation must use this function, not Python `len()`. Confidence: high -- both existing scripts already handle this correctly.

3. **Doc structure assumptions.** The script assumes the document has no content at index 1 that would interfere with the insertion, and that the paragraph style inheritance behavior is consistent. The existing approach of resetting to NORMAL_TEXT first mitigates this. Confidence: high.

4. **Bullet preset names.** The preset names (`BULLET_DISC_CIRCLE_SQUARE`, `BULLET_CHECKBOX`) are Google Docs API constants. They must match exactly. Confidence: high -- these are well-documented.

5. **gws-safe dry-run flow.** Write operations return exit code 2 with a nonce on stderr. The `execute_batch_update()` function from existing scripts handles this pattern. Confidence: high -- proven in production.

6. **Entry existence check.** Determining if today's entry exists requires reading the document. This is the one API call the create path makes before deciding its course. If the cache already has today's entry, the cache can be trusted (it's populated after every successful create/update). Confidence: medium -- cache could be stale if the doc was edited externally.

7. **Sub-skill sequencing.** In the update path, `update-task-list` must run before `update-open-tickets` because both update the cache, and the second run uses the cache updated by the first. If they ran in parallel, cache writes could conflict. Confidence: high -- sequential execution is correct.

---

## Recommendations

1. **Build `bin/update-daily-log` as a Python script** following the same structure as the existing `bin/update-open-tickets` and `bin/update-task-list` scripts.

2. **Extract shared code into a module** (`lib/daily_log_common.py` or similar) since all three scripts share: `utf16_len`, `load_json_file`, `run_cmd`, `get_cache_db`, `parse_doc_entries`, `parse_date_from_heading`, `parse_entry_sections`, `parse_subsections`, `SECTION_MAP`, `extract_section_text`, `execute_batch_update`, `update_cache`. This avoids tripling the maintenance surface.

3. **Reuse cached fetch functions** from `update-task-list` for calendar events and tasks, and from `update-open-tickets` for open tickets. The create path needs the same data.

4. **Handle the random fact** with a static placeholder or a pre-generated data file. Do not call the LLM for this.

5. **Defer archive functionality** to a separate script (`bin/archive-daily-log`) in a later phase. The daily create/update path is the high-frequency optimization target.

6. **Add `--auto-confirm` flag** matching the existing scripts' pattern for non-interactive execution.

7. **Update the SKILL.md** to delegate to the script, matching the pattern already established by the task-list and open-tickets skills.
